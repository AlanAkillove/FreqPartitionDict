# 频率分区字典：面向倾斜工作负载的混合数据结构

---

## 摘要

本研究对 FreqPartitionDict 进行了全面的性能评估，这是一种基于访问频率动态分区数据的混合字典结构。通过在具有不同倾斜特征的合成工作负载上进行大量基准测试，我们证明了所提出的结构在高度倾斜的访问模式下比标准有序映射快达 5.8 倍，同时在均匀分布下保持竞争性性能。我们的分析揭示了工作负载特征、缓存容量配置和整体系统性能之间关系的关键见解。

---

## 1. 引言

现代计算系统越来越多地面临具有显著时间和空间局部性的数据访问模式。从 Web 服务器日志到数据库查询跟踪，实证研究一致表明，一小部分数据项占据了不成比例的大量访问请求——这一现象可以用 Zipf 分布很好地建模 [1]。

传统的字典实现面临一个基本的权衡：哈希表提供 O(1) 平均情况查找，但缓存局部性差且迭代顺序不可预测；平衡树提供 O(log n) 查找和有序迭代，但存在指针追踪开销。两种结构都没有显式利用访问频率信息来优化倾斜工作负载。

本工作介绍了 **FreqPartitionDict**，一种混合字典，它将频繁访问的项保存在高性能的基于哈希的"热区"中，同时将冷数据降级到基于树的"冷区"。我们评估了它在不同工作负载特征下的性能，并提出了实用的部署指南。

---

## 2. 设计与实现

### 2.1 架构概述

FreqPartitionDict 采用两级存储层次结构：

**热区 (Hot Zone)。** 以 `std::unordered_map` 实现，为高频项提供 O(1) 平均情况查找。热区作为固定大小的缓存运行，采用基于频率的淘汰策略。

**冷区 (Cold Zone)。** 以 `std::map`（红黑树）实现，为不频繁访问的项的长尾提供 O(log n) 查找。有序结构在需要时支持高效的范围查询。

**晋升策略 (Promotion Policy)。** 项在冷区累积访问计数；达到可配置阈值（默认：3 次访问）后，它们迁移到热区。该阈值代表了晋升敏捷性和热区稳定性之间的权衡。

**淘汰策略 (Eviction Policy)。** 当热区达到容量时，访问频率最低的项被降级到冷区。原始实现使用线性扫描（O(H)），而优化变体采用最小堆实现 O(log H) 淘汰。

### 2.2 复杂度分析

| 操作 | 热区 | 冷区 | 总体 |
|----------------|---------------|----------------|--------------|
| 查找（命中） | O(1) | — | O(1) |
| 查找（冷区） | O(1) | O(log n) | O(log n) |
| 插入 | O(1) | O(log n) | O(log n) |
| 淘汰（基础） | O(H) | O(log n) | O(H) |
| 淘汰（优化） | O(log H) | O(log n) | O(log H) |

*表1：核心操作的时间复杂度，其中 H 表示热区容量。*

## 3. 实验方法

### 3.1 测试环境

所有实验在具有以下配置的系统中进行：

- **CPU：** Intel Core i7-12700H (14 核, 20 线程)
- **内存：** 16 GB DDR5-4800
- **编译器：** GCC 13.2 with `-O3` optimization
- **操作系统：** Windows 11 (WSL2)

### 3.2 工作负载特征

我们采用 Zipf 分布来建模访问倾斜度。Zipf 概率质量函数定义为：

$$P(k) = \frac{k^{-\alpha}}{\sum_{i=1}^{N} i^{-\alpha}}$$

其中 α 控制倾斜度：α = 0 产生均匀分布，而更高的值将访问集中在更少的项上。我们评估 α ∈ {0.5, 0.8, 1.0, 1.2, 1.5} 以覆盖从几乎均匀到高度集中的访问模式范围。

### 3.3 对比基线

- **std::unordered_map：** 标准哈希表实现
- **std::map：** 红黑树实现
- **FreqPartitionDict：** 具有不同热区容量的提议结构

## 4. 结果与分析

### 4.1 访问倾斜度的影响

![图1：Zipf 对比](https://github.com/AlanAkillove/FreqPartitionDict/raw/main/results/figures/fig1_zipf_comparison.png)

*图1：查找延迟作为 Zipf α 参数的函数。较低的值表示更好的性能。参考线显示标准容器的性能。*

图1 揭示了工作负载倾斜度与结构性能之间的显著关系。随着 α 从 0.5 增加到 1.5，查找延迟减少了约 89%（从 2307 ns 到 252 ns）。这一显著改进源于两个因素：

1. **增加的热区命中率：** 更高的 α 将访问集中在更少的项上，增加了请求数据位于 O(1) 热区中的概率。

2. **减少的冷区压力：** 随着访问的唯一项减少，冷区大小有效缩小，提高了其缓存效率。

在 α = 1.5 时，FreqPartitionDict 实现了在 std::unordered_map 的 2 倍范围内的延迟（244 ns 对 129 ns），同时提供了基于频率的数据组织的额外好处。

### 4.2 热区容量配置

![图2：容量扩展](https://github.com/AlanAkillove/FreqPartitionDict/raw/main/results/figures/fig2_capacity_scaling.png)

*图2：(a) 查找延迟和 (b) 命中率作为热区容量的函数。容量 64 处的拐点表示缓存行为中的相变。*

图2 展示了热区容量与系统性能之间的关键关系。面板 (a) 显示了跨容量的相对稳定的延迟，而面板 (b) 揭示了一个更微妙的故事：

**H = 64 处的相变。** 命中率从 23.6% (H = 32) 急剧增加到 56.8% (H = 64)，代表了 2.4 倍的改进。这个拐点表明，对于评估的工作负载（1000 项，α = 1.0），大约 6.4% 工作集大小的热区容量捕获了大部分热数据。

**收益递减。** 超过 H = 64 后，进一步的容量增加产生递减的收益：命中率从 56.8% 提高到 67.8% (H = 128)，但以双倍的内存消耗为代价。

### 4.3 操作类型对比

![图3：操作对比](https://github.com/AlanAkillove/FreqPartitionDict/raw/main/results/figures/fig3_operation_comparison.png)

*图3：跨操作类型和数据结构的延迟对比。条形图上方的注释值表示以纳秒为单位的平均延迟。*

图3 中的操作级分析提供了几个见解：

**查找操作。** 在 Zipf 访问下（左组），FreqPartitionDict (1283 ns) 比 std::map (316 ns) 快 4.1 倍，但由于晋升/淘汰开销，它仍然比 std::unordered_map (129 ns) 慢。

**均匀访问。** 使用均匀分布的请求（中间组），优势减弱：FreqPartitionDict (3504 ns) 变得竞争力较低，因为热区无法有效捕获工作集局部性。

**插入。** 所有结构显示可比较的插入性能，FreqPartitionDict (495 ns) 介于 std::unordered_map (129 ns) 和 std::map (316 ns) 之间。

## 6. 讨论

### 6.1 何时使用 FreqPartitionDict

我们的评估提出了三个主要用例：

1. **高度倾斜的工作负载 (α > 1.2)：** 该结构实现接近哈希表的性能，同时提供对分析和缓存管理有用的基于频率的组织。

2. **内存受限环境：** 当无法缓存完整工作集时，热区充当智能准入过滤器，优先处理高价值项。

3. **混合访问模式：** 具有明显热数据和冷数据阶段的工作负载受益于自动适应，无需手动调优。

### 6.2 配置指南

基于实证结果，我们建议：

- **热区容量：** 从预期工作集大小的 6-10% 开始。监控命中率并相应调整。
- **晋升阈值：** 默认值 (3) 对大多数工作负载都适用。增加以获得更稳定的热区；减少以实现更快的适应。
- **淘汰策略：** 当 H > 64 时使用基于堆的优化 (O(log H))；线性扫描足以应对较小的容量。

### 6.3 局限性与未来工作

当前的局限性包括：

- **无持久化：** 该结构不支持磁盘备份存储或崩溃恢复。
- **单线程设计：** 并发访问需要外部同步（尽管提供了线程安全变体）。
- **固定阈值：** 静态晋升阈值无法适应变化的工作负载特征。

未来的方向包括自适应阈值调整、多级热区 (L1/L2) 以及与持久内存技术的集成。

---

## 7. 结论

FreqPartitionDict 证明，将访问频率感知纳入字典设计可以为倾斜工作负载带来显著的性能优势。我们的评估显示，在真实访问模式下，比有序映射快达 5.8 倍，与哈希表具有竞争性性能。该结构的简单性和可配置性使其适合集成到缓存系统、数据库索引和分析管道中。

这项工作的关键见解是，工作负载特征非常重要：没有单一数据结构在所有场景中都占主导地位，但适应观察到的模式的混合方法可以弥合理论复杂性和实际性能之间的差距。
